================================================================================
PROJEKTO FAILŲ GIDAS (PROJECT FILE GUIDE)
================================================================================

Šis gidas suteikia išsamią svarbiausių projekto failų apžvalgą. Jis skirtas tiems,
kurie prižiūrės, modifikuos arba testuos „nowcasting“ (trumpalaikio prognozavimo)
sistemą. Sistema geba apdoroti mišraus dažnio makroekonominius duomenis, griežtai
testuoti modelius be duomenų nutekėjimo (angl. look-ahead bias) bei lyginti skirtingas
prognozavimo metodologijas (DFM, BVAR, regresijas, MIDAS, giliojo mokymosi modelius)
naudojant lygiagretų hiperparametrų optimizavimą.

--------------------------------------------------------------------------------
KAIP NAUDOTIS ĮRANKIU (ŽINGSNIS PO ŽINGSNIO)
--------------------------------------------------------------------------------
Norint sėkmingai paruošti ir naudoti „nowcasting“ sistemą, rekomenduojama laikytis
šios veiksmų sekos:

1. APLINKOS KONFIGŪRAVIMAS.
   - Sukurkite PostgreSQL duomenų bazę (pvz., `nowcast_db`).
   - Užpildykite `.env` failą (nurodykite DB_URL, API raktus, jei reikia).

2. SCHEMOS SUKŪRIMAS.
   - Pirmas žingsnis – inicializuoti lenteles duomenų bazėje.
   - PALEIDIMAS: `python create_schema.py`
   - *Pastaba: Ši komanda ištrina visus esamus duomenis bazėje.*

3. DUOMENŲ ATSIUNTIMAS (INITIAL INGEST).
   - Užpildykite tuščią bazę istoriniais duomenimis iš visų tiekėjų (Eurostat, FRED ir kt.).
   - PALEIDIMAS: `python run_ingest.py --mode initial`

4. REGULIARŪS DUOMENŲ ATNAUJINIMAI.
   - Kasdieniam ar savaitiniam naujausių duomenų papildymui naudokite:
   - PALEIDIMAS: `python run_updates.py`

5. DUOMENŲ PARUOŠIMAS MODELIAVIMUI.
   - Sujunkite visas laiko eilutes į vieną panelį, sutvarkykite stacionarumą ir transformacijas.
   - PALEIDIMAS: `python scripts/data_preparation.py --mode common_frequency --target BVP_SA`
   - Tai sukurs apdorotus failus (pvz., .parquet) `data/processed/` kataloge.

6. EKSPERIMENTŲ VYKDYMAS.
   - Paleiskite modelių hiperparametrų paiešką ar backtesting'ą.
   - PALEIDIMAS: `python experiments/run_tabular_search.py --engine staged` (arba kiti runner'iai).

7. REZULTATŲ ANALIZĖ (DASHBOARD).
   - Naudokite vizualizacijos įrankį prognozėms ir metrikoms stebėti.
   - PALEIDIMAS: `streamlit run dashboard.py`

--------------------------------------------------------------------------------
KAIP VEIKIA ŠIS PROJEKTAS? (BENDRA DUOMENŲ IR PROCESŲ EIGA)
--------------------------------------------------------------------------------
Prieš pradedant analizuoti atskirus failus, svarbu suprasti, kaip duomenys keliauja
projekto viduje (koks yra bendras "pipeline"):

1. DUOMENŲ ATSISIUNTIMAS (ETL FAZĖ).
   - Skriptai (`run_updates.py`, `run_ingest.py`, `load_new_source.py`) parsisiunčia
     žalius duomenis iš Eurostat, FRED, Yahoo Finance ir kitų API.
   - Šie neapdoroti duomenys (.csv/.json) išsaugomi `data/raw/` kataloge, o vėliau
     sukeliami į PostgreSQL duomenų bazę.
   - KAIP PALEISTI: `python run_updates.py`

2. DUOMENŲ PARUOŠIMAS MODELIAMS.
   - Duomenų bazėje esančios DB lentelių eilutės nėra iš karto tinkamos modeliams. Tuomet
     naudojamas pagrindinis paruošimo skriptas. Jis sujungia datas (Pivot), sinchronizuoja ir
     užpildo trūkstamas reikšmes, kad modeliai turėtų su kuo mokytis.
   - Išvestis – sugeneruoti apdoroti failai `data/processed/` kataloge (pvz. parquet formatais).
   - KAIP PALEISTI: `python scripts/data_preparation.py --mode common_frequency --target BVP_SA`

3. MODELIŲ TRENIRAVIMAS IR TESTAVIMAS.
   - Paruošti duomenys paduodami į eksperimentų runner'ius iš `experiments/` katalogo.
   - Runner'iai treniruoja modelį pagal visą istoriją slenkančiu (backtesting) rėžimu:
     * `staged` / `grid` varikliai ieško geriausių nustatymų.
     * `evaluate` variklis testuoja 1 konkretų modelį iš nurodyto .json failo.
   - Rezultatai, metrikos ir prognozės išsaugomi `data/forecasts/`.
   - SISTEMOS TĘSTINUMAS (Checkpointing): Visi eksperimentai palaiko `--resume` funkciją, 
     leidžiančią tęsti nutrūkusį skaičiavimą arba praleisti jau atliktus testus.
   - KAIP PALEISTI: `python experiments/run_tabular_search.py --models lightgbm --engine staged`

4. VARTOTOJO SĄSAJA (DASHBOARD).
   - Naršyklėje veikiantis dashboard'as apjungia visas projekto dalis: skaito failus iš `data/forecasts/` 
     grafikams brėžti, stebi sistemos būklę naršyklėje tiesiai iš PostgreSQL.
   - KAIP PALEISTI: `streamlit run dashboard.py`

File: scripts/data_preparation.py
------------------------------------
Paskirtis: Automatizuoja duomenų valymą, transformaciją ir agregavimą į tvarkingus
parquet panelinius failus.
Vaidmuo pipeline: Tai pirmasis etapas. Modeliai naudoja būtent čia sugeneruotus
parquet failus. Šis failas užtikrina, kad eilutės būtų stacionarios ir tvarko
trūkstamas reikšmes.
Numatytieji parametrai:
    mode: PrepMode = PrepMode.COMMON_FREQUENCY
    min_obs: int = 8
    fill_internal_limit: int = 3
    impute_ragged_edges: bool = False
    ragged_edges_limit: int = 4

    # Fast Mode & Scalability
    fast_mode: bool = False
    allowed_frequencies: str = "D,W,M,Q,A"
    seasonality_mode: str = "full"      # 'full', 'fast', 'metadata_only'
    report_level: str = "full"          # 'full', 'compact'
    prefilter_max_series: Optional[int] = None

    # Common-frequency options
    target_freq: str = "ME"
    daily_weekly_rule: str = "mean"

    # Stationarity (common-frequency)
    use_yoy: bool = True
    kpss_alpha: float = 0.05
    max_diffs: int = 2

    # Stationarity (mixed-frequency stationarity (lighter)
    mf_max_diffs: int = 1
    mf_use_log_diff: bool = True      # use log-diff(1) instead of YoY for MF

    # New Requirements
    min_span_years: float = 14.0      # Minimum span between first and last valid observation

    # Execution
    use_multiprocess: bool = False
    n_workers: int = min(os.cpu_count() or 4, 16)

    # Checkpointing & Scalability
    chunk_size: int = 1000
    resume_from_checkpoint: bool = False
    debug: bool = False
Ką reiškia šie numatytieji parametrai:
- `mode`: `common_frequency` visiškai peragreguoja visą duomenų masyvą į atitinkamą
 `target_freq` (pvz. ME - mėnesinį) dažnį.
 `mixed_frequency` palieka originalius dažnius tokiems modeliams kaip MIDAS.
- `ragged_edges_limit` / `impute_ragged_edges`: Tvarko „ragged edges“ problemą išmaniai
 užpildant oficialių publikacijų atsilikimų spragas failo pabaigoje.
- `fast_mode`: Įjungus jį apdorojimas paspartėja dešimtimis kartų praleidžiant gilius
 stacionarumo testus ar sunkų sezoniškumo derinimą. Idealus pasirinkimas dideliems
 atrankos grid'ams.
- `use_yoy`, `kpss_alpha`, `max_diffs`: Stacionarizacijos variklis. Visi duomenys
 automatiškai persukami per KPSS testą, o jei jie ne stacionarūs – sistemingai taikomi
skirtumai (iki max_diffs), kol atitinka kpss_alpha lygį.
- `min_span_years`: Jeigu po trūkstamų duomenų išmetimo eilutė turi mažiau nei
nurodyta metų istorijos, ji atmetama, siekiant modeliams palikti tik aukštos
kokybės ilgalaikes eilutes.
- `resume_from_checkpoint` bei `chunk_size`: Atminties (RAM) pritrūkimo sprendimas.
Didžiuliai laiko eilučių masyvai yra skaidomi į mažus blokus (chunks). Kiekvienas
apdorotas blokas fone išsaugomas per `.parquet` tarpinėje direktorijoje. Jei apdorojimas
netyčia „nulūžta“, su `resume_from_checkpoint` komanda jį galima tęsti praleidžiant jau
turimus duomenis.
Saugu keisti: `fast_mode` greitam ir resursų neeikvojančiam derinimui. `n_workers` bei
`chunk_size` norint pritaikyti kodą kompiuterio sisteminiams pajėgumams.
Keisti atsargiai: `min_span_years`. Jį sumažinus pamaitintų duomenų kiekis
padidės, bet sutrumpės laiko rėžyje, kas drastiškai iškreipia backtesting metrikas.
Paleidimo pavyzdžiai:
python scripts/data_preparation.py --mode common_frequency --fast-mode
python scripts/data_preparation.py --resume-from-checkpoint
Svarbios pastabos: Naudojant perskirstytą atminties arhitektūrą
("streaming/checkpointing"), dabar ši „data_preparation“ sistema gali apdoroti
absoliučiai gigantišką masyvą duomenų vengiant "Out Of Memory" (OOM) klaida.

File: nowcasting/evaluation/backtester.py
------------------------------------
Paskirtis: Atlieka out-of-sample forward-rolling istorinius modelių vertinimus.
Vaidmuo pipeline: Tai griežtas evaluatorius, užtikrinantis, kad
nebūtų duomenų nutekėjimo. Jis iteratyviai plečia arba stumia treniravimo langą,
apmoko transformacijas ir selector’ius, treniruoja modelį ir tuomet vertina
prognozes už treniravimo ribų.
Numatytieji parametrai:
--initial_train_periods: 120
--step_size: 1
--window_type: 'expanding'
--eval_mode: 'common_frequency'
Ką reiškia šie numatytieji parametrai: Vertinimas pradedamas po pirmų 120 periodų
treniravimo. Tada langas juda po 1 periodą, vis plėsdamas treniravimo aibę,
o ne naudodamas fiksuotą slenkantį langą.
Saugu keisti: step_size (pvz., nustatyti 3, jei nori mažiau prognozavimo taškų
ir greitesnio vykdymo), initial_train_periods (didinti, jei modeliai
pradžioje per lengvai persimoko).
Keisti atsargiai: window_type (perjungus į 'rolling', sudėtingi modeliai
gali neturėti pakankamai istorinių duomenų stabiliems parametrams;
DFM/BVAR dažnai mėgsta expanding).
Naudojimo pavyzdys (naudojamas skriptų viduje):
bt = RollingBacktester(initial_train_periods=120, step_size=3)
eval_df = bt.backtest(model, X_panel, y_target, feature_selector=selector)
Svarbios pastabos: Automatiškai seka feature reduction
etapus (raw -> transformed -> selected -> used_by_model) ir perduoda šią
informaciją runner’iams.

File: nowcasting/models/dfm.py
------------------------------------
Paskirtis: Įgyvendina standartinius ir mišraus dažnio Dynamic Factor Models.
Vaidmuo pipeline: Tai parametrinis baseline modelis,
kuris iš šimtų eilučių išskiria latentinius bendrus faktorius ir prognozuoja BVP
naudodamas Kalman Filter.
Numatytieji parametrai:
--k_factors: 1
--factor_order: 1
--maxiter: 200
Ką reiškia šie numatytieji parametrai: Daroma prielaida, kad yra 1 latentinis
ekonominis faktorius su AR(1) dinamika. maxiter nurodo maksimalų EM algoritmo
iteracijų skaičių.
Saugu keisti: k_factors (pvz., bandyti 2 ar 3, kad modelis sugautų skirtingus
ekonominius sektorius), maxiter (sumažinti iki 50 greitam testavimui).
Keisti atsargiai: factor_order. Didesnės reikšmės gali stipriai destabilizuoti
state-space kovariacines matricas ir sukelti singular matrix problemas.
Naudojimo pavyzdys:
model = DynamicFactorNowcast("BVP", k_factors=2, factor_order=1)

File: nowcasting/models/bvar.py
------------------------------------
Paskirtis: Įgyvendina Frequentist VAR ir Bayesian Vector Autoregressions.
Vaidmuo pipeline: Tiesiogiai modeliuoja bendrą kintamųjų dinamiką laike.
Naudoja Minnesota prior, kuris stipriai sutraukia dideles sistemas link random walk,
taip sumažindamas over-parameterization.
Numatytieji parametrai:
--mode: 'bvar'
--lags: 1
--lambda1: 0.1 (bendras shrinkage)
--lambda2: 0.5 (cross-variable shrinkage)
--lambda3: 1.0 (lag decay)
--max_vars: 30
Ką reiškia šie numatytieji parametrai: lambda1 reguliuoja bendrą suspaudimo stiprumą;
kuo mažesnė reikšmė, tuo stipresnis traukimas link nulio. lambda2 reiškia,
kad kitų kintamųjų istorija laikoma mažiau patikima nei paties target istorija.
max_vars uždeda kietą viršutinę ribą sistemos dydžiui, kad būtų išvengta atminties
problemų ir singular matrix klaidų.
Saugu keisti: lags (didinimas nėra labai brangus, bet mažina laisvės laipsnius),
lambda1 (galima bandyti net 0.01 tankioms panelėms).
Keisti atsargiai: max_vars. Jei nustatysi > 50, analitinis posterior skaičiavimas gali
smarkiai išauginti RAM naudojimą arba sugesti.
Naudojimo pavyzdys:
model = BVARNowcast("BVP", mode="bvar", lags=2, lambda1=0.2, max_vars=40)
Svarbios pastabos: Prognozė vykdoma rekursyviai žingsnis po žingsnio į priekį,
neatnaujinant X_test konteksto.

File: nowcasting/models/midas.py
------------------------------------
Paskirtis: Įgyvendina U-MIDAS, kuris tiesiogiai regresuoja žemo dažnio
target į aukšto dažnio lagus.
Vaidmuo pipeline: Pagrindinis mišraus dažnio modelis. Leidžia nenaudoti
agregavimo, nes aukšto dažnio lagai paverčiami į plokščius feature stulpelius.
Numatytieji parametrai:
--n_lags: 1
--freq_ratio: None (automatinis nustatymas)
--fill_strategy: 'zero'
--regression_model: 'elasticnet'
Ką reiškia šie numatytieji parametrai: Sukuriamas 1 atitinkamas aukšto dažnio lagų
langas kiekvienam žemo dažnio žingsniui. Aukšto ir žemo dažnio santykis
apskaičiuojamas automatiškai. Trūkstamos galinės reikšmės pakeičiamos 0.0.
Saugu keisti: fill_strategy (pvz., bandyti 'ffill_then_zero'), n_lags (iki 3 ar 4).
Keisti atsargiai: regression_model. Nors palaikomi 'ridge', 'lasso', 'linear',
būtent 'elasticnet' dažnai apsaugo nuo kolinearumo, kuris atsiranda
dėl lag stacking.
Naudojimo pavyzdys:
model = MIDASNowcast("BVP", n_lags=3, regression_model="elasticnet")
Svarbios pastabos: Jei duomenys tankūs ir feature skaičius viršija stebėjimų
skaičių, paprasta tiesinė regresija gali labai stipriai overfit’inti.

File: nowcasting/models/bridge_equation.py
------------------------------------
Paskirtis: 2 etapų mišraus dažnio integracija. Pirmiausia užpildo ragged edges,
tada vykdo agreguotą regresiją.
Vaidmuo pipeline: Klasikinis ekonometrinis nowcasting metodas. Tvarko „staggered
releases“, prognozuodamas pagrindinius mėnesinius rodiklius iki ketvirčio ribos
naudodamas AR modelius.
Numatytieji parametrai:
--ar_lags: 1
--agg_rule: 'mean'
--regression_model: 'elasticnet'
Ką reiškia šie numatytieji parametrai: Naudoja AR(1), kad užpildytų trūkstamas
aukšto dažnio vietas. Mėnesinės reikšmės agreguojamos į ketvirčius paprastu vidurkiu.
Saugu keisti: agg_rule (kai kuriems duomenims gali labiau tikti 'sum' arba 'last').
Keisti atsargiai: Nėra itin pavojingų parametrų, tačiau reikia stebėti vykdymo laiką,
jei naudojamos labai didelės feature aibės kartu su sunkesniais ElasticNetCV gridais.
Naudojimo pavyzdys:
model = BridgeEquationNowcast("BVP", ar_lags=2, agg_rule="last")
Svarbios pastabos: Automatiškai naudoja StandardScaler, kad suvaldytų skirtingus
feature mastelius prieš CV regresijas.

File: nowcasting/models/ml_regression.py

Paskirtis: ML baziniai modeliai, veikiantys su vienodo dažnio paneliniais duomenimis.
Vaidmuo pipeline: Pateikia patikimą, jokiomis griežtomis prielaidomis neparemtą funkcionalumą
sureguliuoti algoritmą nuo X kintamųjų iki Y tikslo (BVP). Naudoja LightGBM ir

Numatytieji parametrai:

model = LightGBMNowcast("BVP", n_estimators=300, min_child_samples=10)

File: nowcasting/models/tactis_wrapper.py
------------------------------------
Paskirtis: Giliojo mokymosi "Transformer" (TACTiS2) modelio apvalkalas,
pritaikantis jį prie laiko eilučių prognozavimo.
Vaidmuo pipeline: Tai "State-of-the-art" daugiakintamasis, "attention" mechanizmu
paremtas modelis galingiausioms ir tiksliausioms prognozėms.
Numatytieji parametrai:
history_length: 24
epochs: 20
num_samples: 100
skip_copula: False
Ką reiškia šie numatytieji parametrai: Modelis žiūri į praeitį tik 24 žingsnius.
Treniravimas vyksta tik 20 iteracijų/epochų. Prognozė ištraukiama skaičiuojant medianą
iš 100 atskirai sugeneruotų Monte-Carlo Copula trajektorijų.
Saugu keisti: skip_copula (nustačius į True, modelis nerealaus greičio testavimui).
batch_size didinimas taip pat pagreitina prognozes esant didelei atminčiai.
Keisti atsargiai: epochs skaičių. TACTiS modelis itin greitai permoksta (overfits)
ant mažų makroekonominių duomenų. num_samples daro logaritminę įtaką atminties poreikiui.
Naudojimo pavyzdys:
model = TACTiSNowcastWrapper("BVP", epochs=5, num_samples=30, skip_copula=True)
Svarbios pastabos: Wrapperyje yra speciali tik jam sukurta prognozavimo logika.
Jam maitinamas apčiuopiamas ateities kontekstas, tuo pat metu užmaskuojant
tą Y reikšmę, kurią modelis turi atspėti.

Paskirtis: Variational Autoencoder (VAE), veikiantis kaip netiesinis dimensijų mažinimas

Vaidmuo pipeline: Galingesnė alternatyva PCA. Suskaldo milžinišką bruožų

latent_dim: 8

epochs: 30
Ką reiškia šie numatytieji parametrai: Duomenys yra glaudinami "peršaunant" viską per 8
mazgų lauką. Tai kartojama 30 iteracijų.
Saugu keisti: latent_dim.
Keisti atsargiai: Nėra itin rizikingų pakeitimų.
Naudojimo pavyzdys:
model = VAENowcast("BVP", latent_dim=10, epochs=50)
Svarbios pastabos: Treniravimas vyksta dviejose fazėse. Pirma - generuojamas erdvės
"suspaudimas" jį paliekant nevaldomą, po to jį "užšaldant", uždaroma procedūra
regresija skirta testavimui ant užšaldytų latentinių mazgų.


--------------------------------------------------------------------------------
PART 3: EXPERIMENT RUNNERS (GRID, STAGED & EVALUATE)
--------------------------------------------------------------------------------
Projektas turi grupę vienodos architektūros runner skriptų, esančių `experiments/` kataloge:
--run_tabular_search.py (ElasticNet, LightGBM)
--run_midas_search.py (MIDAS)
--run_bridge_search.py (Bridge Equations)
--run_dfm_experiment.py (DFM)
--run_bvar_experiment.py (BVAR, VAR)
--run_tactis_experiment.py (TACTiS Transformer)

Paskirtis: Sistemiškai testuoja hiperparametrų tinklelius arba vertina konkrečius modelius.
Vaidmuo pipeline: Pagrindinis įrankis modelių kokybės matavimui istoriniuose duomenyse.

Bendri bruožai ir varikliai (--engine):
1. staged (Numatytasis): Atlieka 2 pakopų paiešką. Pirma pakopa testuoja viską ant 
   trumpo lango (3 periodų), antra pakopa paima Top K modelius ir testuoja pilnu rėžimu.
2. grid: Testuoja viską po vieną pilnu rėžimu (lėtas, bet išsamus).
3. optuna: Naudoja „Bayesian Optimization“ (TPE sampler), kad automatiškai rastų 
   optimalius parametrus erdvėje.
4. evaluate: Reikalauja `--config-path`. Paima konkretaus modelio nustatymus (pvz. iš 
   .json failo arba nukopijuotus iš CSV) ir atlieka vieną tikslų backtestą.

Kiti svarbūs parametrai:
--resume: Įjungia kontrolinių taškų (checkpoints) sistemą. Jei programa nulūžta 
  skaičiavimų viduryje, paleidus dar kartą su šiuo flag'u, darbas bus tęsiamas 
  nuo ten, kur sustota.
--search-last-n-steps: Apriboja vertinimą tik iki paskutinių N laiko taškų (labai 
  naudinga greitam testavimui). 0 reiškia pilną istoriją.
--n-jobs: Lygiagrečių procesų skaičius. Dauguma modelių saugiai veikia su n-jobs=4 
  arba daugiau, tačiau TACTiS geriau palikti 1.

Visi runner’iai eksportuoja:
- gridsearch_[model].csv — pilnas žurnalas su visų bandymų metrikomis ir hash'ais.
- predictions_[model].csv — geriausios rastos konfigūracijos taškinės prognozės.
- best_config_[model].json — laimėjusio modelio parametrų archyvas, tinkamas 
  vėlesniam vertinimui per `--engine evaluate`.

Svarbios pastabos: Checkpoint sistema naudoja MD5 hash'avimą, todėl net menkiausias 
parametro pakeitimas (pvz. learning_rate nuo 0.1 iki 0.10001) privers sistemą 
perskaičiuoti modelį iš naujo, nes tai bus laikoma nauja konfigūracija.
--------------------------------------------------------------------------------
PART 4: OPTUNA - BAYESIO OPTIMIZAVIMAS
--------------------------------------------------------------------------------
Kai tradicinis tinklelis (Grid Search) tampa per didelis arba per lėtas, projektas
naudoja „Optuna“ biblioteką intelektualiam parametrų parinkimui.

Kaip tai veikia:
Modelių runner'iai naudoja TPE (Tree-structured Parzen Estimator) algoritmą. 
Tai nėra tiesiog aklas bandymas — Optuna mokosi iš ankstesnių bandymų rezultatų 
ir bando tas parametrų kombinacijas, kurios turi didžiausią tikimybę duoti 
geresnį RMSE.

Pagrindiniai parametrai:
--engine optuna: Perjungia runner'į į Bayesian optimizavimo režimą.
--n-trials: Kiek iš viso bandymų atlikti (pvz. 50 ar 100).
--study-storage: Nuoroda į SQLite duomenų bazę (pvz. `sqlite:///optuna.db`), 
  kurioje saugoma visa bandymų istorija. Tai leidžia saugiai tęsti tyrimą 
  net po kompiuterio perkrovimo.
--save-every-n-trials: Kas kiek bandymų išsaugoti progresą į tarpinį CSV failą.

Svarbios pastabos:
- Optuna automatiškai saugo „best_config.json“ su geriausiais rastais parametrais.
- Naudojant `n_jobs > 1` su Optuna, ji naudoja SQLite užraktus (locking), kad 
  keli procesai galėtų saugiai dalintis ta pačia bandymų istorija.
- Jei modelio treniravimas „nulūžta“ dėl klaidos, Optuna tą bandymą pažymi kaip 
  `FAIL` ir eina prie kito, nesustabdydama visos sistemos.

--------------------------------------------------------------------------------
PART 9: DUOMENŲ SAUGOJIMO MECHANIKA IR MIGRACINĖS SISTEMOS
--------------------------------------------------------------------------------

1. Failų išsaugojimo ekosistema (Caching strategija):
- Žali API duomenys (Raw Storage): Žali API atsakymai iš karto išsaugomi į
 `data/raw/` kartu su failo SHA256 kodu (hash). Ingestinimo
  sistema (`run_updates.py`) kiekvieną kartą sutikrina, ar naujai atsisiųstas failas
   tokio paties atspaudo kaip praėjusį kartą. Jei taip, tai lėta duomenų bazės
   papildymo transakcija visiškai ignoruojama/praleidžiama, taip sutaupant resursų.
- Vektorinis/Panelinis duomenų saugojimas (Vector Storage): Sukompiliuoti duomenys
greitam modelių treniravimui skaitomi iš `data/processed/` Parquet formatu, kuris
yra 10-100 kartų greitesnis nei standartinis CSV.
- Modelių ir Prognozių saugojimas (Model Forecasts): `data/forecasts/` kataloge
išsaugomi modelių failai, kuriuos vartotojo sąsaja (`dashboard.py`) tiesiogiai
sinchronizuoja ir realiu laiku atvaizduoja ekrane (be būtinybės dar kartą daryti lėtų
kreipinių į modelį).

2. Duomenų bazės schemos keitimo modelis (Alembic Migrations):
PostgreSQL lentelės nėra kietai įkaltos (angl. hardcoded). Jas seka išorinė „Alembic“
biblioteka, kuri užtikrina, kad pridėjus naują stulpelį į SQL, nereiktų visko trinti
 iš naujo.
KAIP ATNAUJINTI DUOMENŲ BAZĘ PRIDĖJUS STULPELĮ KODE:
`alembic revision --autogenerate -m "Added new column"`
`alembic upgrade head`
Ši migracijų sistema efektyviai eliminuoja poreikį valyti esamus istorinius duomenis
 (`clear_data.py`) norint pakoreguoti schemos struktūrą.

FRED ir kitų šaltinių. Keičiant šį failą galima labai lengvai pridėti naujus rodiklius
be kodo keitimo.

Files: config/sources/*.json
------------------------------------
Paskirtis: Custom šaltinių apibrėžimai (pvz., example_csv.json, world_bank.json).
Vaidmuo pipeline: Naudojami kartu su `load_new_source.py` skriptu kaip šablonai
prijungiant naujus, neeilinius API ar CSV failus per JSON dot-notation konfigūraciją.

File: docker-compose.yml
------------------------------------
Paskirtis: Docker konteinerių orkestravimas.
Vaidmuo pipeline: Leidžia vienos komandos pagalba (`docker-compose up -d`) greitai
pakelti lokalų PostgreSQL duomenų bazės serverį testavimui ar darbui.

File: requirements.txt
------------------------------------
Paskirtis: Python bibliotekų priklausomybės.
Vaidmuo pipeline: Nurodo visas išorines bibliotekas (pvz. pandas, yfinance, sqlalchemy),
kurios reikalingos šiam projektui. Skirtas garantuoti nuoseklumą tarp skirtingų aplinkų
(`pip install -r requirements.txt`).

--------------------------------------------------------------------------------
PART 13: DIRECTORY STRUCTURES & ADDITIONAL MODULES
--------------------------------------------------------------------------------

Files: nowcasting/**/__init__.py
------------------------------------
Paskirtis: Python paketų inicializacija.
Vaidmuo pipeline: `nowcasting/__init__.py`, `evaluation/__init__.py`, `features/__init__.py` ir `models/__init__.py`
paverčia šiuos aplankus importuojamais moduliais. Dažniausiai būna tušti.

File: nowcasting/utils/reproducibility.py
------------------------------------
Paskirtis: Sėklų (seeds) ir atsitiktinumo valdymas.
Vaidmuo pipeline: Užtikrina atkuriamumą užfiksuojant NumPy, PyTorch ir Python
pseudo-atsitiktinių skaičių generatorius. Naudojamas visuose ML / DL modelių
apmokymuose ir iteracijose, siekiant pašalinti atsitiktines variacijas tarp tų
pačių hiperparametrų paleidimų.

Directory: data/raw/ (ir scripts/data/raw/)
------------------------------------
Paskirtis: Atsisiųstų pirminių duomenų kaupykla.
Vaidmuo pipeline: Čia originaliu formatu (dažniausiai CSV arba JSON failais,
pvz. `worldbank_ltu_gdp_...json`) išsaugomi atsakymai iš išorinių API
(Eurostat, FRED ir t.t.). Tai yra pirminis šaltas saugojimas prieš perkeliant viską
pradine forma į PostgreSQL.

Directory: data/processed/
------------------------------------
Paskirtis: Transformuotų ir paruoštų analizei duomenų bazė.
Vaidmuo pipeline: Saugomi galutiniai, agreguoti ir pagal dažnį sulygiuoti panelinių
duomenų failai (dažniausiai `.parquet` arba `.csv`). Šiuos failus pasigamina
`data_preparation.py` skriptas ir juos vėliau modelyje valgo backtesteris.

Directory: data/forecasts/ (ir scripts/data/forecasts/)
------------------------------------
Paskirtis: Modelių prognozių ir metrikų išvestis.
Vaidmuo pipeline: Į šį aplanką nukreipiami eksperimentų (pvz., `run_bvar_experiment.py`)
rezultatai. Čia saugomi atrinktų geriausių architektūrų konfigūracijų failai
(`best_config_*.json`), testavimo laikų žurnalai ir pačių testavimų metrikos
(`*_eval.csv`). Šiuos failus surenka `dashboard.py` vizualizacijai naršyklėje.

Directory: tactis_repo/
------------------------------------
Paskirtis: TACTiS 2 modelio bibliotekos talpykla.
Vaidmuo pipeline: Tai yra TACTiS 2 neuroninis tinklas. Šiame aplankale tiesiog saugomi
failai, susiję vien tik su šia giliojo mokymosi struktūra. Jų funkcionalumas atsiskiria
nuo pagrindinio pipeline, todėl šių atskirų failų detaliai aprašinėti nereikia.
